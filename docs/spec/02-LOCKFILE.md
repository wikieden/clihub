# Spec 02 — clihub.lock.json v1 (stable, v0.6.1)

Generated by `clihub lock` from `clihub.yaml`; consumed by `clihub install
--frozen` and `clihub status`. Pins exact resolved versions so the same
lockfile reproduces the same toolchain.

## Shape

```json
{
  "version": 1,
  "generatedAt": "2026-05-29T00:00:00.000Z",
  "source": "clihub.yaml",
  "clihub": "1.4.0",
  "tools":   { "claude-code": { "version": "2.1.150", "method": "npm" } },
  "skills":  { "superpowers": { "tools": ["claude-code", "codex"] } },
  "mcp":     { "context7": {} },
  "plugins": { "some-plugin": {} }
}
```

## Fields

| Field | Type | Meaning |
|---|---|---|
| `version` | `1` | lockfile format version |
| `generatedAt` | ISO 8601 | when generated (informational; excluded from any signature) |
| `source` | string | the config it was generated from (`clihub.yaml`) |
| `clihub` | string | clihub version that wrote it |
| `tools` | map id → `{version, method?}` | resolved tool versions; `version` is a concrete version or `latest` when no precise pin exists |
| `skills` | map id → `{tools[]}` | which CLIs each skill targets |
| `mcp` | map id → `{}` | MCP servers (presence only in v1) |
| `plugins` | map id → `{}` | plugins (presence only in v1) |

## Compliance (clihub status)

For each tool: detect installed version, compare to `tools[id].version`.

- match → `ok`
- mismatch → `drift`
- not installed → `missing`
- `latest` pin (no precise version) → `unlocked` (cannot assert drift)

`clihub status --strict` exits non-zero on any `drift` or `missing`.

## Atomicity

Written via a `.tmp` file + atomic rename. Read returns `undefined` on
ENOENT so callers can fall back to `clihub.yaml`.
