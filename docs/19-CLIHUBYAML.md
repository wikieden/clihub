# `clihub.yaml` — declarative project config (Pillar II)

`clihub.yaml` is the source of truth for "what AI coding environment this project needs". Teams check it into git; `clihub apply` brings the local machine in line with it. This document is the schema + behaviour contract for Sprint 8 (v0.5.3).

## 1. Minimum viable example

```yaml
# clihub.yaml — at the root of any git project
version: 1
profile: work                  # auto-switch when entering this dir

tools:
  - claude-code
  - codex

skills:
  - superpowers
  - tdd
  - review

presets:
  - starter

mcp:
  - filesystem
  - github

plugins: []

proxy:
  inherit: true                # use the user's global proxy config
```

`clihub apply` reads this file, diffs against the active state, and converges. `clihub apply --plan` shows the diff without writing.

## 2. Full schema (v1)

```yaml
# Schema version. Bumped only on incompatible changes.
version: 1

# Optional profile override; falls back to ~/.clihub/config.json:defaultProfile.
profile: <string>

# CLIs the project needs installed.
tools:
  - id: claude-code            # short form `claude-code` also accepted
    version: ">=2.0 <3.0"      # semver range; omit for any
    method: npm                # npm | bun | brew | curl
  - codex

# Skills to install. Each entry is either an id (resolves via catalog) or a
# git URL / local path (SKILL.md installer).
skills:
  - superpowers
  - id: review                 # explicit form
    tool: claude-code          # narrow to one CLI (default: all supported)
  - source: https://github.com/foo/bar.git    # git URL form
    tool: claude-code
  - source: ./skills/team-style              # local path

# Presets (compose tools+skills+MCP+plugins). Applied after explicit lists.
presets:
  - starter

# MCP servers. Each entry is either an id or an inline definition.
mcp:
  - filesystem                                # catalog id
  - id: github                                # explicit form
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: ${env:GH_TOKEN}   # value substitution
  - id: my-custom                             # inline definition
    command: npx
    args: ["-y", "my-mcp-pkg"]
    env: {}

# Plugins (currently Claude Code only).
plugins:
  - oh-my-claudecode
  - id: ccpi-everything
    branch: main                              # optional non-default branch

# Proxy override for this project. `inherit: true` uses the user's global.
proxy:
  inherit: true
  # or override:
  # https: http://corp.proxy:8080
  # noProxy: localhost,*.corp.example.com

# Custom CA bundle path for this project (overrides global).
caBundle: ./.clihub/corp-ca.pem

# Sharing rules for the active profile (see 18-CONFIG-PROXY-PROFILE.md).
shared: [skills, plugins]
not-shared: [settings.api-keys]

# Lifecycle hooks (run by `clihub apply`). Optional.
hooks:
  pre-apply:
    - "echo applying clihub.yaml"
  post-apply:
    - "git add ~/.claude/settings.json || true"
```

## 3. Resolution rules

### Tools

For each `tools[]` entry:

1. If `version` range supplied → install matching; abort if no match.
2. If CLI already installed at a satisfying version → no-op.
3. Otherwise call the provider's `install({ method, version })`.

### Skills

1. Bare id → resolve via `CatalogLoader.findSkill(id)`; abort if missing.
2. `source:` git URL or local path → SKILL.md installer (already implemented for `clihub skill install <git|path>`).
3. Per-CLI fan-out follows `tool:` if set; else every CLI in `supports[]`.

### Presets

Applied last (after explicit tools/skills/mcp/plugins). A preset can only *add*; it never removes items already declared above.

### MCP

1. Bare id → catalog lookup.
2. Inline `command/args/env` → installed without catalog entry.
3. `env:` values support `${env:VAR}` substitution.

### Plugins

1. Bare id → catalog lookup.
2. `id` + optional `branch` / `ref` overrides git clone behaviour.

## 4. Variable interpolation

Inside any string value, `${env:VAR}` is replaced with `process.env.VAR` at apply time. Missing env → abort with `CLIHUB-E-602` (overridable with `--allow-missing-env`).

`${file:./path.txt}` substitutes file contents (trimmed). Useful for tokens checked into git-crypt'd files.

`${secret:KEY}` reads from the active profile's keychain vault. Never logged.

## 5. `clihub.lock.json`

Generated by `clihub apply` and `clihub install --frozen` consumes it.

```jsonc
{
  "version": 1,
  "generatedAt": "2026-07-15T09:12:48.190Z",
  "source": "clihub.yaml",
  "sourceSha256": "sha256:<hash-of-clihub-yaml>",
  "clihub": "0.5.3",                       // engine version that wrote the lock

  "tools": {
    "claude-code": {
      "version": "2.1.150",
      "method": "npm",
      "installedAt": "2026-07-15T09:12:50.001Z"
    },
    "codex": { "version": "0.133.0", "method": "npm", "installedAt": "..." }
  },

  "skills": {
    "superpowers": {
      "version": "1.4.2",
      "source": "https://github.com/obra/superpowers.git",
      "commit": "abc123def4567890",
      "tools": ["claude-code"]
    }
  },

  "mcp": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "${HOME}"],
      "tools": ["claude-code", "gemini-cli"]
    }
  },

  "plugins": {
    "oh-my-claudecode": {
      "source": "https://github.com/yeachan-heo/oh-my-claudecode.git",
      "commit": "fedcba9876543210",
      "tools": ["claude-code"]
    }
  }
}
```

### `--frozen` semantics

```bash
clihub install --frozen
```

- Refuses if `clihub.lock.json` is missing or `version` is unsupported.
- Errors on drift: any catalog entry whose current SHA differs from the lockfile's pinned SHA aborts with the offending items listed.
- Bypass: `clihub install --frozen --update <id>` re-locks a single entry.

## 6. CLI surface

```bash
clihub init                                  # interactive scaffold; writes clihub.yaml
clihub init --non-interactive --preset starter   # CI-friendly
clihub apply                                 # converge local state to clihub.yaml
clihub apply --plan                          # show diff, write nothing
clihub apply --dry-run                       # alias of --plan
clihub install --frozen                      # apply from clihub.lock.json
clihub lock                                  # regenerate clihub.lock.json without applying
clihub validate                              # parse + check refs; exit non-zero on errors
```

## 7. Plan output format

```
$ clihub apply --plan
clihub.yaml → diff against current state

  tools
    + codex                  (will install via npm, latest)
    = claude-code 2.1.150    (already installed, satisfies ">=2.0 <3.0")

  skills
    + tdd                    → claude-code
    + tdd                    → codex
    ~ review (v1.2 → v1.4)   → claude-code

  mcp
    + filesystem             → claude-code, gemini-cli
    - obsolete-mcp           → removed (not in clihub.yaml)

  plugins
    + oh-my-claudecode       → claude-code

  proxy
    = inherit (effective: http://corp.proxy:8080)

Summary: 5 to add, 1 to update, 1 to remove. Run `clihub apply` to execute.
```

Legend: `+` add, `-` remove, `~` upgrade, `=` no-op.

## 8. Validation rules

- `version` must equal `1` for now.
- `profile` must match `[a-z][a-z0-9-]{0,30}`.
- Every `id` referenced must resolve in the active catalog (`clihub validate` checks this).
- `version` ranges follow [npm semver](https://docs.npmjs.com/cli/v10/using-npm/semver) syntax.
- Unknown top-level keys are warned about but not rejected (forward compat).

## 9. Project layout suggestion

```
my-project/
├── clihub.yaml
├── clihub.lock.json          ← commit this
├── .clihub/
│   ├── corp-ca.pem           ← per-project CA (gitignored if private)
│   └── secrets.env           ← gitignored; sourced into ${env:*}
├── src/
└── ...
```

`.gitignore` template:

```
.clihub/secrets.env
.clihub/corp-ca.pem
```

(Keep these out of git; commit the lockfile.)

## 10. CI usage

```yaml
# .github/workflows/dev.yml
- uses: clihub/setup-action@v1                 # v0.7 deliverable
  with: { version: '0.5.3' }
- run: clihub install --frozen
- run: clihub doctor --json
```

For repositories without the setup action:

```bash
npm install -g @wikieden/clihub@latest
clihub install --frozen
```

## 11. Migration from ad-hoc usage

If you've been using clihub interactively, `clihub init --import-current` snapshots the active profile's installed tools / skills / plugins / MCP into a starting `clihub.yaml`. Tweak as needed, then `clihub apply` is a no-op (or shows the diff if something drifted).

## 12. Error codes (yaml-specific)

| Code | Meaning |
|---|---|
| `CLIHUB-E-600` | `clihub.yaml` not found in the resolution path |
| `CLIHUB-E-601` | YAML parse error |
| `CLIHUB-E-602` | required env var missing during interpolation |
| `CLIHUB-E-603` | catalog id not found during validate / apply |
| `CLIHUB-E-604` | lockfile drift on `install --frozen` |
| `CLIHUB-E-605` | unsupported `version` in clihub.yaml |
| `CLIHUB-E-606` | hook command exit non-zero (aborts apply unless `--continue-on-error`) |

## 13. Out-of-scope (this design)

- Profile-per-branch (git branch → profile) — possible v0.7 follow-up.
- Re-exports / includes (`extends: another.yaml`) — defer until pain emerges.
- Sub-project nesting beyond one `clihub.yaml` per repo.
- GUI editor — TUI editing is fine for v1.0.
